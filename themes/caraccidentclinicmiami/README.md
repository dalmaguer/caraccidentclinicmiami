# Car Accident Clinic Miami — Block Theme

Custom FSE block theme for [caraccidentclinicmiami.com](https://caraccidentclinicmiami.com).

---

## Tailwind CSS Build

Tailwind is compiled locally from source — no CDN, no plugin required. The compiled file (`assets/css/tailwind.css`) is loaded statically in the `<head>` on the frontend and in the block editor.

### Prerequisites

- Node.js ≥ 18
- npm ≥ 9
- WP-CLI available at `/usr/local/bin/wp` (used by the DB content dump)

### Install dependencies

Run once after cloning or after pulling changes to `package.json`:

```bash
npm install
```

### Build for production

Dumps DB content, then compiles and minifies Tailwind CSS:

```bash
npm run build
```

Run this whenever you:
- Add new Tailwind classes to any pattern file (`patterns/*.php`)
- Add new Tailwind classes to template or part files (`templates/*.html`, `parts/*.html`)
- Publish new pages/posts that contain Tailwind classes not already covered by the safelist

### Watch mode (development)

Dumps DB content once, then watches all theme files and recompiles on every save:

```bash
npm run dev
```

> **Note:** Watch mode does not re-dump DB content on each change. If you add new classes to a page via the block editor and save, run `npm run build` once to pick them up.

### Dump DB content only

Re-generates `bin/content-scan.html` from the WordPress database without rebuilding CSS:

```bash
npm run dump
```

## How class coverage works

Tailwind scans three sources before compiling:

| Source | Scanned by |
|---|---|
| `patterns/*.php` | File scan (always up to date) |
| `templates/*.html`, `parts/*.html` | File scan (always up to date) |
| Block editor classes saved to pages/posts in the DB | `bin/content-scan.html` (generated by `npm run dump`) |

In addition, a **safelist** in `tailwind.config.js` pre-compiles the most common utility classes (flex, grid, gap, margin, padding, width, height, typography, colors, shadows, etc.) regardless of whether they appear in any file. This catches any class you add directly in the editor without needing a rebuild.

### When do you need to run `npm run build`?

| Scenario | Action needed |
|---|---|
| Added classes to a pattern/template/part file | `npm run build` |
| Added an obscure class directly on a page in the editor | `npm run build` (covers the DB) |
| Added a class already in the safelist | Nothing — it's already compiled |
| Deployed to production | Always run `npm run build` before deploying |

## File structure

```
caraccidentclinicmiami/
├── assets/
│   └── css/
│       ├── tailwind.src.css   ← Tailwind input (base / components / utilities)
│       └── tailwind.css       ← Compiled output — enqueued on frontend + editor
├── bin/
│   ├── dump-content.sh        ← WP-CLI script: dumps DB post content for scanning
│   └── content-scan.html      ← Generated — gitignored, rebuilt on every build
├── package.json               ← build / dev / dump scripts
└── tailwind.config.js         ← Content paths + safelist
```

## Adding new Tailwind classes — quick reference

**In a pattern file** (`patterns/my-pattern.php`)
→ Just add the class. Run `npm run build` when done.

**Directly on a page in the block editor**
→ If the class is in the safelist, no action needed.
→ If it's a custom or obscure class, run `npm run build` after publishing.

**New utility not in the safelist**
→ Either add it to the `safelist` array in `tailwind.config.js`, or just run `npm run build` after saving the page.

## Deactivating WindPress

WindPress is no longer needed for Tailwind on the frontend — `tailwind.css` is already loading. To safely remove it:

1. Verify the site looks correct with WindPress **deactivated** (not deleted) first.
2. Check the homepage cover block — WindPress added a `has-windpress-slate-50-background-color` class. Replace it with `bg-slate-50` or remove it if the background is set via `theme.json` instead.
3. Once confirmed, delete the WindPress plugin.
